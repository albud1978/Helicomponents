# Мультиагентный шаблон для Cursor AI

> Универсальный набор конфигурационных файлов для организации мультиагентного workflow в Cursor IDE.
> Основан на best practices 2025–2026 (Anthropic, Google ADK, Cursor), проверен на реальном проекте.

---

## Быстрый старт

1. Скопируйте содержимое `.cursor/` в корень вашего проекта.
2. Адаптируйте файлы агентов под ваш стек и домен (см. раздел «Кастомизация»).
3. Проверьте, что `90_multiagent_workflow.mdc` применяется глобально (`alwaysApply: true`).

│   ├── project.mdc                 # Обвязка auto-apply (указатель на правила)
│   ├── 00_global_always.mdc        # Глобальные правила (SOLID/DRY/KISS, порядок работы)
│   ├── 30_capsules_context.mdc     # Scoped rule для капсул (auto-apply при docs/*_capsule.md)
│   └── 90_multiagent_workflow.mdc  # Workflow (роли, handoff, governance, enforcement, RLM)
├── skills/                          # Опционально: доменные/технологические skills проекта

│       └── SKILL.md                # Пример SQL/DB guard skill (адаптировать под вашу СУБД)

    ├── validator.md                # Валидатор (проверка результатов)

## Архитектурные принципы

### 1. Иерархия, а не плоская структура

**Плохо:** множество «равных» агентов без координатора — ведёт к хаосу, дублированию работы и agent sprawl.

**Хорошо:** чёткая иерархия с Orchestrator наверху:
- Orchestrator ставит задачи и контролирует прогресс.
- Subagents (coder, reviewer, validator) работают в своих зонах.
- Человек утверждает ключевые решения через governance-гейты.

### 2. Sequential Pipeline как базовый паттерн

Последовательный пайплайн — самый надёжный и отлаживаемый паттерн:

```
Анализ → Исследование → Реализация → Ревью → Валидация → Governance → Docs → Capsule
```

- Один агент за раз, строгий порядок.
- Следующий агент вызывается только после получения Handoff от предыдущего.
- Параллельные и итеративные паттерны — только после зрелости sequential.

### 3. Явные Hand-off артефакты

Агенты передают друг другу не «сырые ответы», а структурированные пакеты:

```
Handoff:
- UserGoal: дословная цель запроса
- TraceID / PlanStepID: идентификаторы трассы и шага плана
- RiskTier / RiskReasons: уровень риска и причины
- Changes: что сделано
- Facts / Assumptions: проверенное vs непроверенное
- EvidencePack / ComplianceChecklist: доказательства и policy-check
- HumanGateRequired / ApprovalGate: нужен ли human gate и его статус
- OpenQuestions: что осталось неясным
- NextOwner: кто следующий
- GraphUpdate: требуется ли запись в Domain Graph
- Agent KG logging: какие записи сделаны в Agent KG на шаге
- Risks: ключевые риски
- DriftCheck / ProcessInsights: anti-drift и процессные выводы
```

### 4. Human-in-the-Loop (Governance)

Определённые действия требуют явного подтверждения человека:
- Архитектурные решения
- Изменения схем данных
- Удаление файлов/модулей
- Новые публичные API

### 5. Графы и память (write-through)

- **Agent KG** — операционный граф, пишется постоянно на каждом переходе фазы (`dispatch`, `phase_start`, `handoff`, `close`).
- **Domain Graph** — архитектурный граф, обновляется только при архитектурных изменениях и только после явного approval человека.
- Для `medium/high-risk` оркестратор фиксирует `ArchitectureImpactProposal` (`required|not_required`) с фактическим обоснованием.
- Закрытие high-risk workflow возможно только после `governance-compliance` verdict и синхронизации документации (`docs-curator`).

### 6. Observability

- Каждый агент логирует reasoning-шаги в Handoff.
- Решения трассируемы: от цели до конкретного действия.
- При дрейфе контекста — контрольная точка и сверка с целью.
- Локальный дэшборд строится из источников по умолчанию: `config/agent_kg.json` (и опционально `config/capsules_manifest.json`) для прозрачного контроля работы агентов.
- Минимальные секции дэшборда: текущий статус исполнения workflow и последние взаимодействия агентов.

### 7. Enforcement (технические guardrails)

Декларативных правил ("оркестратор не кодит") недостаточно — LLM может их "забыть" при длинном контексте. Шаблон включает технические слои защиты:

1. **Prompt reinforcement** (`orchestrator_guard.py`) — governance-напоминание инжектируется при каждом промпте через `beforeSubmitPrompt`.
2. **User communication audit** (`user_comm_audit.py`) — метаданные коммуникации и approval-hint для human-on-the-loop трассировки.
3. **Policy self-check** — CRITICAL маркеры в правилах и обязательная проверка пути перед Write/StrReplace/Shell.
4. **Post-factum audit** (`audit_code_edit.py`) — `afterFileEdit` логирует правки в зонах кодеров.
5. **Pre-close guard** (`pre_close_guard.py`) — блокирует `--close-workflow` без обязательных handoff и trace-полей.
6. **Connection fail guard** (`connection_fail_*`) — fail-safe блокировка tool-вызовов при проблемах подключения.
7. **Delegation-depth guard (policy)** — вложенный найм subagent'ов блокируется правилами (разрешен только 1 уровень делегирования).

> **Важно**: Cursor hooks не передают роль агента (orchestrator vs subagent) в payload. Поэтому deny по зонам кодеров в `preToolUse` заблокирует и самих кодеров. Используемый подход: reinforcement + observability + policy verdict.

## Локальные дэшборды (Agent KG Viewer)

Дэшборд — это read-only визуализация состояния мультиагентного контура без изменения SSoT.

- **Источник данных (по умолчанию):** `config/agent_kg.json` (обязательно), `config/capsules_manifest.json` (для RLM-метрик).
- **Генерация (по умолчанию):** `python3 tools/agent_kg_viewer/build_agent_kg_viewer.py`
- **Выход (по умолчанию):** `tools/agent_kg_viewer/index.html`
- **Что должно быть видно минимум:**
  - `Текущий статус исполнения` — активные workflow, фаза, владелец, последнее событие.
  - `Последние взаимодействия агентов` — хронологическая лента handoff/context.

Рекомендуемые дополнительные секции: `WorkflowTrace`, `GovernanceGates`, `RLMReuse`.

> Если в вашем проекте другие пути/имена, замените их в генераторе и в чеклисте внедрения.

## Роли агентов

### Orchestrator (обязательная роль)
- **Цель:** декомпозиция задач, координация, governance.
- **Кодинг запрещён** — только планирование и управление.
- Для исследования кодовой базы использует встроенный `explore` subagent.
- Принимает Handoff от всех агентов, решает о завершении или итерации.

### Coder (1 или несколько специализаций)
- **Цель:** реализация конкретных изменений в коде.
- Работает строго в своей зоне, не расширяет scope самовольно.
- Получает чётко сформулированные задачи от Orchestrator.

> **Совет:** при большом проекте разделите на специализированных кодеров
> (например, `coder-backend`, `coder-frontend`, `coder-infra`).

### Reviewer (рекомендуемая роль)
- **Цель:** проверка качества — «вторая пара глаз».
- Не пишет код, только оценивает и даёт фидбэк.
- Выносит вердикт: «принять», «вернуть на доработку», «эскалировать человеку».

### Validator (опциональная роль)
- **Цель:** фактическая проверка результатов (тесты, данные, SQL).
- Работает с фактами, не с рассуждениями.
- Результат: PASS / FAIL с доказательствами.

### Analyst / Researcher (опциональная роль)
- **Цель:** сбор контекста, данных и подготовка спецификаций.
- НЕ принимает архитектурных решений — только готовит базу.
- Передаёт результаты кодерам и оркестратору.

### Capsule Builder (опциональная роль)
- **Цель:** фиксация знаний и контекста для передачи между чатами.
- Формирует структурированные капсулы после завершения задач.
- Обновляет `config/capsules_manifest.json` — индекс всех капсул.

### Governance / Compliance (рекомендуемая роль control-plane)
- **Цель:** policy/risk/compliance проверка перед закрытием high-risk задач.
- Не пишет код; выносит вердикт `approve|reject|escalate`.
- Проверяет traceability (`TraceID`, `PlanStepID`, `ApprovalGate`) и соответствие scope.

### Docs Curator (рекомендуемая роль документации)
- **Цель:** синхронизация `docs/**`, `README.md`, `docs/changelog.md` с принятыми решениями.
- Не пишет код; фиксирует только документальные артефакты и их трассировку.
- Для high-risk задач handoff от docs-curator обязателен перед закрытием workflow.

## Контекстные капсулы (RLM)

Капсулы решают проблему «10M+ токенов контекста»: вместо загрузки всего проекта агент читает индекс → выбирает 1-2 релевантные капсулы (~100-200 строк) → работает в фокусе.

### Как это работает

```
Агент получает задачу
  → читает config/capsules_manifest.json (~30 строк)
  → выбирает нужную капсулу по domain/covers
  → читает капсулу (~150 строк)
  → работает в узком фокусе вместо 50K+ строк проекта
```

### Формат капсулы (8 секций)

Каждая капсула — файл `docs/<domain>_capsule.md` с 8 обязательными секциями:

1. **Scope** — что трогаем / не трогаем (ограничение контекста)
2. **Invariants** (≤12) — ключевые инварианты со ссылкой на SSoT
3. **Decisions** (≤7) — архитектурные решения с обоснованием
4. **Impact Paths** — граф влияния (файл → что зависит)
5. **Validation Proof** — как проверяется корректность
6. **Risks** (≤7) + Mitigations — риски и их митигации
7. **Open Questions** (≤7) — нерешённые вопросы
8. **Pointers** (≤15) — ссылки на файлы для глубокого погружения

### Иерархия источников (строгая)

```
Данные/Конфиги (SSoT) → Капсулы (контекст для агентов)
```

Капсулы — **read-only проекции**, не источник истины. При расхождении — верить SSoT.

### Манифест (`config/capsules_manifest.json`)

```json
{
  "version": 1,
  "capsules": [
    {
      "id": "auth",
      "path": "docs/auth_capsule.md",
      "domain": "Аутентификация, сессии, JWT",
      "covers": ["src/auth/**", "middleware/auth.ts"],
      "updated": "2026-02-10"
    }
  ]
}
```

### Внедрение капсул

1. Создайте `config/capsules_manifest.json` с пустым массивом `capsules`
2. После завершения первой значимой задачи — попросите capsule-builder создать капсулу
3. Капсулы накапливаются постепенно, покрывая основные домены проекта

## Кастомизация под ваш проект

### Шаг 1: Определите зоны кода

Разделите проект на зоны ответственности. Примеры:

| Проект | Зона 1 | Зона 2 | Зона 3 |
|--------|--------|--------|--------|
| Web-приложение | `backend/**` | `frontend/**` | `infra/**` |
| ML-проект | `models/**` | `data/**` | `api/**` |
| Монорепо | `packages/core/**` | `packages/web/**` | `packages/cli/**` |

### Шаг 2: Назначьте агентов на зоны

- По одному кодеру на зону (или объединяйте мелкие зоны).
- Один reviewer может покрывать несколько зон.
- Validator привязан к инструментам проверки (тесты, SQL, API).

### Шаг 3: Определите governance-гейты

Что в ВАШЕМ проекте требует одобрения человека? Типичные кандидаты:
- Изменения в API-контрактах
- Миграции БД
- Изменения CI/CD
- Новые зависимости
- Удаление кода/файлов

### Шаг 4: Настройте Handoff-шаблон

Базовый шаблон работает для большинства проектов. Добавьте поля под ваш домен:
- `Migration`: есть ли миграции (для проектов с БД)
- `APIChanges`: затронуты ли публичные API
- `PerformanceImpact`: влияние на производительность

### Шаг 5: Выберите модель

В frontmatter каждого агента указана модель. Рекомендации:
- **Orchestrator / Reviewer** — используйте самую сильную доступную модель (требуется глубокое понимание).
- **Coder** — сильная модель с хорошей генерацией кода.
- **Validator / Analyst** — можно использовать менее дорогую модель (задачи формализованы).

## Антипаттерны (чего избегать)

### 1. Flat-архитектура без оркестратора
Множество «равных» агентов без координатора приводит к:
- Дублированию работы
- Конфликтам при параллельном редактировании
- Избеганию сложных задач (каждый берёт простое)

### 2. Полная автономия без governance
Отсутствие approval-точек означает:
- Необратимые действия без контроля
- Человек не понимает, что происходит
- Потеря доверия к системе

### 3. Тихое подавление ошибок
Агенты не должны скрывать проблемы. Каждая ошибка должна быть:
- Зафиксирована в Facts (или в Assumptions при невозможности проверки)
- Эскалирована через Handoff
- Видна человеку

### 4. Бесконечные итерации
Без лимита итераций агент может бесконечно переделывать задачу.
Рекомендация: **максимум 3 итерации**, затем — эскалация человеку с отчётом.

### 5. Agent sprawl
Слишком много ролей усложняет систему. Начинайте с минимума:
- **Минимум:** Orchestrator + Coder + Reviewer (3 роли)
- **Стандарт:** + Validator + Analyst + Governance + Docs Curator (7 ролей)
- **Максимум:** + Capsule Builder + специализированные кодеры (8+ ролей)

---

## Лимит итераций и эскалация

При недостижении результата за N итераций:

1. Orchestrator **останавливает работу**.
2. Формирует **отчёт для человека** (Handoff с полным описанием).
3. **Эскалация** — дальнейшие действия определяет человек.

Агент НЕ имеет права продолжать работу после достижения лимита.

---

## Разрешение конфликтов между агентами

Приоритеты:
1. **Безопасность > Функциональность**
2. **Источник истины > Предположения агента**
3. **Решения человека > Решения оркестратора**
4. **Архитектурные правила > Удобство реализации**

При конфликте:
- Reviewer выносит вердикт
- При серьёзных расхождениях — эскалация человеку
- Решение фиксируется в логах/артефактах
