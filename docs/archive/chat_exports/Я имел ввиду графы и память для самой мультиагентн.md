# Я имел ввиду графы и память для самой мультиагентной системы разработки, а не в контексте ABM модели

Ты спрашиваешь именно про **память и графы как инфраструктуру самой мультиагентной ИНЖЕНЕРНОЙ системы** (оркестратор, код‑агенты, ревью‑агенты), а не про доменную модель — сфокусируюсь на этом.

***

## 1. Какие типы памяти используют в мультиагентной разработке

Сейчас в гайдлайнах и фреймворках для agentic‑разработки почти везде выделяют несколько **слоёв памяти**:[^1][^2][^3][^4]

1. **Рабочая / in‑thread memory**
    - Краткоживущая память в рамках одной задачи или pull‑request: история обсуждений, локальный план, текущие файлы.
    - Живёт в контексте конкретного «сеанса» или ветки; после завершения задачи может быть сжата в короткий summary.[^5][^3]
2. **Кросс‑сеансовая / cross‑thread memory**
    - Общие знания о проекте и командных решениях: архитектурные решения, стандарты, важные баги и их фиксы, «как здесь принято делать Х».[^2][^3][^1]
    - Это то, к чему имеют доступ **все** агенты на проекте (и новые агенты тоже).
3. **Персонифицированная память агентов/пользователей**
    - Память конкретного агента или разработчика: предпочтения, стиль, привычные паттерны; используется для персонализации подсказок и планов.[^6][^2]
    - Обычно отделена неймспейсами (user‑level, team‑level, project‑level).
4. **Системная память решений (decision traces)**
    - Журналы: какие решения и почему приняли агенты (какой план выбрали, какие файлы трогали, что отклонили).[^4][^1][^2]
    - Это основа наблюдаемости и последующей донастройки.

***

## 2. Shared memory между агентами разработки

### 2.1. Общие блоки памяти / shared memory blocks

Фреймворки для мультиагентных систем (вроде Letta и др.) предлагают **общие блоки памяти**, к которым подключают нескольких агентов:[^7][^2][^6]

- Каждый агент имеет свой приватный блок (личная рабочая память) и один или несколько **shared‑блоков** для команды/проекта.
- Когда один агент пишет в shared‑блок (например, план задач или статус работы над файлом), остальные читают это как «истину текущего момента».[^7]

Практики:

- Разделение на **read‑only** и **read‑write** блоки, чтобы не все могли править базовые решения.[^2][^7]
- Использование неймспейсов: `project:helicomponents`, `team:backend`, `agent:orchestrator`, чтобы избежать кросс‑помех.[^2]


### 2.2. Shared memory как контекст‑служба

Google и крупные игроки описывают отдельный **MemoryService** или Context Service, к которой агенты ходят за контекстом, а не хранят всё в себе:[^8][^1][^4]

- Агент завершает задачу → MemoryService индексирует результаты (векторно, по ключевым словам, по графу).
- Другой агент при старте:
    - делает `load_memory` по проекту/файлу/issue;
    - получает только релевантные куски (а не весь лог).[^6][^4]

Это заменяет «context stuffing» (когда в каждый вызов модели тащат всю историю) на **memory‑based workflow**: память живёт отдельно, агенты запрашивают только нужное.[^1][^8][^4]

## 3. Графы для самой мультиагентной среды разработки

Здесь речь не о домене, а о **графе разработки**: задачи, файлы, агенты, статусы.

### 3.1. Context graph / project graph

В современных платформенных гайдах выделяется понятие **context graph**:[^9][^10][^11]

- Узлы:
    - файлы, модули, сервисы;
    - задачи/тикеты, фичи, баги;
    - агенты (planner, coder, reviewer), люди;
    - сессии/PR, релизы.
- Рёбра:
    - «зависит от», «меняет», «связан с issue», «ответственный агент», «ревьюер» и т.п.[^11][^9]

Практики:

- Контекст‑граф определяет **форму задачи**: допустимые пути, критические точки (где обязателен ревью/chekpoint), возможные исполнители.[^9][^11]
- Оркестратор ходит по графу, чтобы понять:
    - какой файл/модуль затронуть;
    - какие другие файлы/сервисы подтянуть;
    - кто (какой агент) сейчас лучший кандидат.[^10][^9]


### 3.2. Knowledge graph как backbone для команды агентов

Для agentic‑разработки всё чаще используют **knowledge graph проекта**:[^10][^1][^6]

- Он хранит:
    - архитектурные решения (ADR);
    - контракты между сервисами;
    - исторические баги и их фиксы;
    - паттерны/анти‑паттерны, принятые в команде.
- Агенты (planner, coder, reviewer) при работе **сначала** обращаются к графу, а уже потом — к сырым файлам и коду.[^6][^10]

Бенефиты:

- Меньше «разрыва мозга» между агентами — они опираются на одни и те же связи и понятия.
- Легче внедрить governance: граф — это «официальная правда», которая переопределяет галлюцинации модели.[^10]

## 4. Практики context engineering для мультиагентной разработки

Современные гайды по context engineering сводят всё к трём ключевым задачам:[^12][^5][^1][^6]

1. **Write context (правильно писать память):**
    - Не вываливать «сырые» логи, а сохранять **структурированные объекты**: План, Решение, Дифф, Ошибка, Метрики.
    - Делать периодические summary, чтобы сжать историю (иначе shared memory зарастёт шумом).[^1][^6]
2. **Select context (правильно читать память):**
    - При старте задачи агенты не поднимают «всё подряд», а используют фильтры/поиск по: файлу, issue, типу задачи, дате, владельцу.[^4][^6]
    - Для сложных задач — комбинируют графовый контекст (что связано) и векторный поиск (детали внутри связанных узлов).[^4][^10]
3. **Isolate context (не путать потоки):**
    - Каждый агент видит **ровно тот** кусок общей памяти, который нужен его роли:
        - planner — граф задач и архитектурные решения;
        - coder — файлы, спеки и прошлые диффы;
        - reviewer — изменения, тесты, политики качества.[^8][^6]
    - Это уменьшает конфликты и «сплетение» разных задач.

## 5. Инструменты взаимодействия агентов поверх памяти/графа

Сами каналы взаимодействия опираются на эту память, а не заменяют её:[^1][^6][^4]

- **Task objects в общей памяти.**
    - Задачи представлены как объекты (узлы/документы) со статусами, владельцами, связями.
    - Handoff = смена владельца/статуса + запись reasoning в этот объект.
- **Decision traces и reasoning logs.**
    - Для каждой крупной задачи:
        - план (какие шаги, почему);
        - выбор архитектуры/решения;
        - причины отклонённых вариантов.
    - Ложится в системную память и привязано к узлам графа (файлы/PR/issue).[^2][^1]
- **Context graphs для диалога и маршрутизации.**
    - Граф состояний процесса разработки (аналог state machine для workflow), где узлы — этапы (plan, implement, review, merge, post‑mortem).[^11][^9]
    - Оркестратор переводит задачу по этому графу, меняя активного агента и активный контекст.
- **Общий namespace для команд/проектов.**
    - Память делится по уровням: org → проект → репо → ветка → задача.[^2]
    - Это даёт правильные «границы видимости» для агентов: код‑агент одного проекта не лезет в память другого.

***

## 6. Что это значит для твоей мультиагентной системы разработки

Если примени это к твоему стеку (мультиагентная разработка ABM/PBM‑системы, Cursor/IDE‑агенты):

- **Shared memory** — это не только файлы в репо, но и отдельный слой:
    - журнал задач/решений (agent decision log);
    - общий «проектный граф» (связи между файлами, симуляциями, валидациями, отчётами);
    - краткие summary прошлых кампаний и изменений.
- **Граф** — это карта **разработки**, а не физики:
    - узлы: модули, конфиги, пайплайны, сценарии симуляций, тест‑сьюты, тикеты, агенты;
    - рёбра: «меняет», «зависит», «проверяет», «запускает», «ответственен за».
- **Оркестратор** не таскает с собой весь контекст, а каждый раз:
    - просит Memory/Graph‑службу дать «окно» контекста по текущей задаче;
    - пишет назад только сжатое и структурированное (планы, диффы, статусы).

- предложить конкретную **схему project‑graph’а** для мультиагентной разработки (какие типы узлов и связей завести);
- показать на словах, как через этот граф проходят твои Planner / Coder / Reviewer / Governance‑агенты в одном типичном цикле работы над фичей.
