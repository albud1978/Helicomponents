# Superset: multi-user API + dashboard-as-code (draft architecture)

## Цель
Организовать единый Superset-инстанс на одной рабочей машине так, чтобы:
- пользователи работали через API и имели **свои черновики**;
- пользователь мог делиться доступом к своим объектам (точечно/группами);
- изменения в BI шли как `dashboard-as-code` (Git-артефакты), без постоянной пересборки Docker;
- пользователь **не мог изменять чужие наработки** без явного grant.

## Что фиксируем как принципы
1. **Один runtime, много пространств владения**:
   - один контейнерный Superset runtime;
   - ownership и доступы разделяются RBAC/RLS, а не отдельными контейнерами на пользователя.
2. **Build редко, import часто**:
   - Docker image пересобирается только при изменении версии Superset/плагина/системных зависимостей;
   - пользовательский BI-контент меняется через API export/import bundle.
3. **Draft-first workflow**:
   - личные draft-объекты пользователя;
   - затем pull request / review;
   - затем promote в shared space.
4. **API как основной интерфейс автоматизации**:
   - сервисные токены (или machine users), а не shared admin-учётка;
   - аудит всех операций создания/изменения/публикации.

## Логическая модель доступа

### Сущности
- `User` — человек или сервисный агент (Cursor AI client).
- `Workspace` — область владения (personal/team/shared).
- `Asset` — dataset/chart/dashboard.
- `AccessPolicy` — права чтения/изменения/публикации.

### Минимальные роли
- `bi_draft_author`: может создавать/редактировать только свои draft-ассеты.
- `bi_reviewer`: читает чужие draft, может оставлять review, без force-edit.
- `bi_publisher`: может продвигать approved-ассеты в shared.
- `bi_reader`: read-only в shared.
- `bi_api_client`: сервисная роль с ограниченным набором API-операций.

## Поток изменений (dashboard-as-code)
1. Пользователь создаёт/обновляет draft через API в personal workspace.
2. Экспортирует bundle в Git-ветку.
3. PR + автоматические проверки (валидность bundle, обязательные поля, naming policy, линтер SQL).
4. После approval — импорт в shared namespace (promote).
5. Присвоение/обновление ACL для целевой аудитории.

## Изоляция пользовательских наработок
- Владение объектом связывается с owner/team tag.
- В shared используются отдельные publish-правила (только через promote).
- Запрет прямой правки shared-ассетов авторами draft без роли `bi_publisher`.
- Для API-клиентов: короткоживущие токены, scope-only permissions, rate limits.

## Операционная схема

### Контур runtime
- Superset + Postgres metadata + Redis в Docker Compose.
- reverse proxy перед Superset (TLS, allowlist, basic WAF/rate limiting).
- async worker/beat включаются при росте нагрузки.

### Контур управления контентом
- Git как источник версии BI-артефактов.
- Bundle export/import как стандартный механизм переноса.
- Чёткое разделение:
  - `personal/*` — пользовательские драфты;
  - `team/*` — командные заготовки;
  - `shared/*` — только опубликованные.

## Non-goals (пока не делаем)
- Авто-деплой в production без ручного gate.
- Прямой runtime patch в контейнере без фиксации в Git.
- Права "всем всё" для ускорения старта.

## Риски и меры
- **Риск:** API-токены с избыточными правами.
  - **Мера:** отдельные сервисные роли + ротация + аудит.
- **Риск:** конфликт импортов (перетирание чужих объектов).
  - **Мера:** namespace по workspace + promote-only в shared.
- **Риск:** дрейф между UI-изменениями и Git.
  - **Мера:** обязательный export в PR как часть Definition of Done.

## Definition of Done для рабочего процесса
- Пользователь может создать draft dashboard через API.
- Пользователь может дать read доступ к draft выбранным ролям/пользователям.
- Без роли publisher нельзя изменить shared-объект.
- Любая публикация в shared проходит через Git PR и импорт.
- Пересборка образа не нужна для контентных изменений.
