# Пользовательские интерфейсы: ввод сценариев → вывод → выгрузка

> Статус: **исследование архитектуры** (не реализация). Версия 2026-06-23.
> Контекст: после ускорения ETL+load (107s → ~24s, см. `4dac335`, `5ccb152`) и
> возможности выбирать любую дату-срез из DWH мы переходим к пользовательским
> доработкам. Цель документа — зафиксировать варианты архитектуры и дать рекомендацию.

## 1. Что хотим (3 интерфейса)

1. **Ввод** — выбор даты-среза + заполнение сценариев и переменных перед прогоном.
2. **Вывод** — визуализация результата (есть сегодня в Superset BI).
3. **Выгрузка** — результат в удобные файлы Excel / PDF.

Пожелание: **не плодить инструменты** — в идеале один вход для пользователя.

## 2. Ключевой инсайт: «выбор даты» — это ДВА разных понятия

Это центральная развилка, которая снимает большую часть путаницы.

| | A. BI-фильтр (есть сегодня) | B. Триггер прогона (нужно построить) |
|---|---|---|
| Что выбираем | `version_date` среди **уже посчитанных** прогонов | `report_date` среза DWH, по которому **ещё не было** прогона |
| Где | Superset native filter (`version_date_ddmmyyyy`) | новый управляющий слой |
| Действие | фильтрация готовых строк в ClickHouse | запуск ETL extract → sim (FLAME GPU) → load |
| Стоимость | мгновенно | минуты + нужна CUDA-GPU |
| Результат | перерисовка чарта | появление НОВОГО `version_date` в витринах |

Superset «выбор любой даты» сегодня = **A**. Пользовательская задача «выбрать дату
из хранилища и задать сценарий» = **B**. Superset из коробки **не умеет B** —
native-фильтр только фильтрует то, что уже лежит в витринах. Это и есть главная
причина, почему «замкнуть всё на Superset» нетривиально.

## 3. Где сейчас живут входные переменные (разбросаны по 4 источникам)

Сценарий прогона сегодня НЕ описывается одним объектом. Параметры размазаны:

| Источник | Переменные | Кто меняет |
|---|---|---|
| **CLI оркестратора** (`orchestrator_limiter_v8.py`) | `--version-date`, `--end-day` (горизонт, дн.), `--max-steps`, `--version-id`, `--drop-table` | инженер вручную |
| **DWH report_date** (`dwh_loader.py --report-date`) | какой срез AMOS тянем | инженер вручную |
| **Config JSON** (`config/transitions/*.json`) | пороги переходов, квоты ремонтных линий, `rtc_execution_order` | редко, через Git |
| **Excel мастер-данных** | `Economics.xlsx` (стоимости ремонта/ferry — экономический гейт), `Program_AC` (target Mi-8/Mi-17 = MP4), `Status_Components` (начальные SNE/PPR/статусы) | подготовка датасета |

Физические факторы сценария (из README): начальное состояние (SNE, PPR, repair_days),
темпы налёта (MP5), программа полётов (MP4 target), сроки ремонтов (repair_time, очередь),
ресурсные лимиты (LL/OH/BR), экономические пороги (ferry/repair).

**Вывод:** чтобы пользователь «заполнял сценарий», нужен новый слой —
**манифест сценария** (один JSON/YAML), который пайплайн умеет читать и
проецировать на CLI-флаги / JSON-оверрайды / значения Excel. Сегодня такого
слоя нет; это отдельная работа, не зависящая от выбора UI.

## 4. Скрытая сложность №1: оркестрация GPU-задач

Симуляция требует `conda env cuda13` + FLAME GPU и идёт минутами. Любой UI,
запускающий прогон, по сути **ставит задачу в очередь на GPU-воркер**:

- UI (Superset/веб) и GPU-раннер могут жить на разных машинах.
- Нужны: постановка задачи, статус/прогресс, идемпотентность (partition
  `(version_date, version_id)` уже даёт per-slice replace), защита от
  параллельных конфликтующих прогонов.
- Это нужно при ЛЮБОМ выборе UI. Минимальный вариант — таблица задач в
  Postgres/ClickHouse + воркер-демон, дергающий существующие CLI-скрипты.

## 5. Что Superset может и чего не может

| Потребность | Superset нативно | Комментарий |
|---|---|---|
| Вывод/дашборды (#2) | ✅ сильная сторона | гистограммы, Gantt-плагин, KPI — уже есть |
| BI-фильтр по дате (A) | ✅ есть | native filter `version_date_ddmmyyyy` |
| Форма ввода сценария (#1) | ❌ нет | нет «форм, запускающих бэкенд-джоб» |
| Триггер прогона (B) | ❌ нет | нужен внешний/кастомный слой |
| Экспорт CSV из чарта | ✅ есть | только CSV, по одному чарту |
| Экспорт **Excel** (#3) | ⚠️ нет нативно | многолистный отчёт — только через наш `openpyxl`-код |
| Экспорт **PDF** (#3) | ⚠️ скриншот | dashboard → PNG/PDF через headless-chrome; это картинка, не табличный отчёт |

Текущий конфиг: `EMBEDDED_SUPERSET: False`, `ENABLE_TEMPLATE_PROCESSING: True`
(Jinja в датасетах — можно параметризовать SQL через URL-параметры).

У нас уже есть зрелый Excel-экспорт на `openpyxl`/`pandas` (`export_engine_repair_dwh.py`,
`dwh_*_export.py`, `05_export_mp5_excel.py`). **PDF в проекте нет** (нет reportlab/
weasyprint). `xlsxwriter` импортируется, но отсутствует в `requirements.txt`.

## 6. Архитектурные опции

### Опция A — «Всё в Superset» (кастомный Flask-blueprint)

Superset — это Flask-приложение; в него можно добавить кастомные blueprint/views.

- **Ввод:** кастомная страница внутри Superset с формой сценария → пишет манифест,
  ставит задачу GPU-воркеру.
- **Вывод:** нативные дашборды (как сейчас).
- **Выгрузка:** кастомный view, вызывающий наши `openpyxl`-экспортёры (+ PDF через
  reportlab/weasyprint).
- ➕ Один URL, один логин, одно развёртывание.
- ➖ Глубокая связка с внутренностями Superset → трение при апгрейдах. Тревожный
  сигнал уже в README: Gantt-кастомизации наносятся **runtime-патчем контейнера**,
  не являются Git-артефактом и теряются при пересборке. Углублять кастомизацию
  Superset = множить такую хрупкость.

### Опция B — Тонкий control-plane + embedded Superset (рекомендуется)

Небольшое веб-приложение (FastAPI+мини-фронт или Streamlit) как «пульт»:

- **Ввод:** нативная форма — date-picker доступных `report_date` из DWH, виджеты
  сценарных переменных (горизонт, квоты, экономика, программа). Пишет манифест
  сценария, ставит задачу GPU-воркеру, показывает прогресс.
- **Вывод:** **встраивает** существующий дашборд Superset (Embedded SDK / guest-token,
  включить `EMBEDDED_SUPERSET`), отфильтрованный на новый `version_date`. Superset
  остаётся «движком визуализации», а не отдельным инструментом, которым жонглирует
  пользователь.
- **Выгрузка:** кнопка → наши `openpyxl`-экспортёры + PDF. Полный контроль формата.
- ➕ Один вход для пользователя; Superset остаётся ванильным/обновляемым; чистое
  разделение ответственности (пульт владеет вводом/джобами/выгрузкой, Superset —
  только просмотр).
- ➖ Больше движущихся частей (свой сервис + guest-token + воркер). Новая зависимость
  (веб-фреймворк) — в проекте её сейчас нет.

### Опция C — Статус-кво (CLI + Superset раздельно)

Ввод/выгрузка через CLI-скрипты, просмотр в Superset. Это «несколько инструментов»,
от чего пользователь явно хочет уйти. Базлайн для сравнения.

## 7. Рекомендация

**Опция B, уточнённая до B-lite внутри существующего портала** (см. раздел 7d).
Форма ввода живёт как модуль в **закрытой части корпоративного портала**, где уже
есть **Keycloak SSO**. Superset подключается к тому же Keycloak по OIDC → единый вход
для формы и дашбордов без второго логина. Этот репозиторий не содержит UI-кода формы —
он отдаёт наружу **бэкенд-контракт** (API доступных дат / submit job / статус / выгрузка),
а Superset остаётся ванильным движком вывода.

Superset силён в #2 — оставляем его этим; #1 (форма+триггер) живёт в портале,
#3 (Excel/PDF) — наш бэкенд-эндпоинт поверх `openpyxl`/WeasyPrint. Опция A
(встраивать всё в Superset) отвергнута: платим связкой с Superset-internals
без выгоды, т.к. портал уже даёт оболочку и auth.

Независимо от UI, сначала нужны два слоя-фундамента (живут в ЭТОМ репозитории за API):

1. **Манифест сценария** — единый JSON/YAML, проецируемый на CLI/JSON/Excel (раздел 3).
2. **Оркестрация GPU-задач** — таблица задач + воркер поверх существующих CLI (раздел 4).

Предлагаемая последовательность (фазы, для будущего обсуждения):
`Ф0` манифест сценария → `Ф1` job-раннер поверх CLI → `Ф2` бэкенд-API (даты/submit/статус) →
`Ф3` форма в портале + permalink/redirect на Superset → `Ф4` Excel/PDF выгрузка →
`Ф5` (опц.) переход redirect → embedded Superset на общей Keycloak-сессии.

## 7a. Детальное сравнение A vs B (по запросу)

Зафиксированный контекст решений: глубина сценариев — **полная** (квоты/экономика/
программа), PDF — **настоящий табличный отчёт**. Это смещает баланс: и форма ввода, и
выгрузка становятся существенными Python-компонентами, что играет в пользу B.

| Критерий | A: всё в Superset (Flask-blueprint) | B: пульт + embedded Superset |
|---|---|---|
| Точек входа для пользователя | 1 URL, 1 логин | 1 URL (пульт), Superset скрыт в iframe |
| Форма ввода сценария | кастомный Flask view + Jinja/JS внутри Superset | нативная форма веб-фреймворка (Streamlit/FastAPI) |
| Сложная форма (per-day MP4/экономика) | бороться с шаблонами Superset | свободный выбор виджетов/загрузки Excel |
| Excel/PDF-выгрузка | view внутри Superset, наши экспортёры | эндпоинт пульта, наши экспортёры |
| Связка с Superset-internals | **высокая** (blueprint, security, версии API) | **низкая** (только guest-token + iframe) |
| Трение при апгрейде Superset | велико: кастом-код может ломаться | мало: Superset ванильный |
| Хрупкость (прецедент в README) | повторяем риск runtime-патчей контейнера | избегаем |
| Аутентификация | переиспользуем Superset auth | **решено: Keycloak SSO портала** (раздел 7d) |
| Развёртывание | один контейнер | +1 сервис (пульт) +1 воркер |
| Новая зависимость | нет веб-фреймворка, но кастом в Superset | веб-фреймворк (сейчас нет в deps) |
| Изоляция отказов | падение формы = риск для Superset | пульт падает независимо от Superset |
| Тестируемость | сложно (внутри Superset) | легко (отдельный сервис) |
| Скорость первого MVP | ниже (освоить Superset-расширение) | выше (greenfield-сервис) |

**Итог сравнения:** при «полной глубине сценариев» и «настоящем PDF» обе тяжёлые
части (#1 и #3) всё равно пишутся на Python вне нативного Superset. A заставляет
встраивать этот Python в Superset (связка + хрупкость) ради «одного URL»; B даёт тот
же «один URL» через iframe, не трогая внутренности Superset. **B остаётся
рекомендацией с большим отрывом** именно из-за выбранной глубины.

## 7b. Последствия «полной глубины сценариев» для слоя-манифеста

Полные оверрайды = манифест должен покрывать три класса параметров с разной природой:

| Класс | Пример | Природа | UX-подход в форме |
|---|---|---|---|
| Скаляры | `end_day`, горизонт, версия | одно число | поле/слайдер |
| Глобальные кнобы | множитель квот линий, общий коэффициент стоимости ferry/repair | скаляр-оверрайд над JSON/Excel | поле + «по умолчанию из датасета» |
| Временные ряды | MP4 target Mi-8/Mi-17 по дням, экономика step-function по датам | per-day массив | **3 уровня:** (1) наследовать из Excel датасета; (2) глобальный сдвиг/множитель; (3) загрузка своего Excel/CSV |
| Срез данных | `report_date` DWH | выбор из доступных | date-picker по списку из DWH |

Рекомендуемая модель: **манифест сценария = base (из датасета `version_date`) + overrides**.
Пользователь не заполняет сотни значений с нуля — он берёт базовый сценарий и меняет
точечно. Манифест (`scenario.yaml`) читается адаптером, который материализует
эффективные `config/*.json` и Excel-входы для конкретного прогона, не трогая каноны в Git.
Нужен **реестр пресетов сценариев** — это и есть продукт для планировщика.

**Зафиксированные оси вариаций пресетов (2026-06-23):**

| Ось вариации | Что меняет | Класс параметра (из таблицы выше) |
|---|---|---|
| Количество плановых поставок | объём входящих поставок компонентов | временной ряд / скаляр |
| Ограничение поставок | кап/лимит на поставки | глобальный кноб |
| Динамический спавн с формированием дефицита | генерация потребности → возникающий дефицит (*реализуется в коде сейчас*) | временной ряд / логика модели |
| Изменение налёта | вектор утилизации (MP5) | временной ряд |
| Изменение количества потребных в ops | потребность в исправных в эксплуатации | временной ряд / скаляр |

Эти пять осей задают приоритет реализации оверрайдов в манифесте. Baseline (данные как
есть из выбранного `report_date`) — нулевой пресет, остальные строятся как точечные
оверрайды поверх него.

## 7c. PDF: настоящий табличный отчёт

Выбран структурированный документ (не скриншот). Технические опции:

- **WeasyPrint** (HTML/CSS → PDF): верстаем Jinja-шаблон, переиспользуем брендбук
  (шрифты Suisse, палитра статусов, даты DD-MM-YYYY). Лучший контроль типографики;
  тянет системные либы (Pango/Cairo).
- **ReportLab**: программная вёрстка таблиц/Flowables; меньше внешних зависимостей,
  больше кода на сложную вёрстку.
- Графики в PDF: либо matplotlib-рендер из тех же ClickHouse-запросов, либо
  PNG-экспорт чартов Superset через API и вставка в отчёт.

Рекомендация: **WeasyPrint + Jinja-шаблон**, разделяющий данные (запрос к витринам) и
вёрстку (брендбук). Добавить в `requirements.txt`: `weasyprint` (+ зафиксировать
`xlsxwriter`, который уже импортируется, но не объявлен). Отчёт собирается тем же слоем,
что и Excel, из одних и тех же запросов к `sim_masterv2_v9_daily` / `sim_repairline_v9`.

## 7d. Интеграция с корпоративным порталом (решающее уточнение)

Форма ввода размещается не как новый отдельный сервис, а как **модуль в закрытой
(аутентифицированной) части существующего корпоративного портала**, где **уже развёрнут
Keycloak**. Портал — отдельная система, **вне этого репозитория**.

### Граница ответственности

```
Портал (отдельная система, Keycloak SSO)        ← UI + auth, НЕ в этом репо
   └─ форма ввода сценария  +  ссылка/embed результатов
            │  HTTP-контракт (API)
            ▼
Этот репозиторий = БЭКЕНД-КОНТРАКТ              ← вычисления и данные
   ├─ GET  /report-dates      доступные срезы DWH
   ├─ POST /runs              манифест сценария → постановка GPU-задачи
   ├─ GET  /runs/{id}         статус/прогресс
   └─ GET  /runs/{id}/export  Excel / PDF
            │
            ▼
Superset                                        ← только вывод
   └─ дашборд, предотфильтрованный на version_date прогона
      (старт: redirect/permalink; цель: общий Keycloak OIDC)
```

### Что это закрывает

- **Auth портала — решён.** Закрытая зона портала за Keycloak даёт единый вход
  для формы «бесплатно».
- **Auth Superset — целевое, не текущее.** На `supersetbi` Keycloak пока **не
  подключён**. Целевое состояние: Superset идёт к тому же Keycloak по OIDC
  (Flask-AppBuilder поддерживает OAuth/OIDC) → тогда нет второго логина ни при
  redirect, ни при embed. До этого redirect на Superset может приводить к его
  собственному логину — это допустимо на старте (см. раздел 8, решение по выводу).
- **Объём резко падает.** Не поднимаем новое приложение со своим логином/оболочкой —
  добавляем страницу в существующий webui. UI-код формы живёт в портале, не здесь.
- **Старт — redirect (permalink)** на нужный `version_date`. Бэкенд от выбора
  redirect/embed не зависит, поэтому переход на embed позже не требует его переделки.
  Бесшовный embed станет дешёвым **после** подключения Keycloak к Superset (общая
  OIDC-сессия); пока этого нет, redirect — правильный старт.

### Граничный нюанс — оформление контракта

Поскольку портал и этот репозиторий — разные системы, **API-контракт между ними должен
быть зафиксирован явно** (OpenAPI-описание эндпоинтов из схемы выше). Это позволяет
команде портала верстать форму независимо от готовности GPU-бэкенда.

## 8. Зафиксированные решения и оставшиеся вопросы

**Решено (2026-06-23):**
- Направление: **B-lite внутри портала** — форма в закрытой части корп-портала,
  Superset для вывода, бэкенд-контракт в этом репо (разделы 7, 7d).
- Глубина сценариев: **полная** — квоты/экономика/программа (раздел 7b).
- PDF: **настоящий табличный отчёт** (раздел 7c, WeasyPrint).
- Auth формы: **решён** — Keycloak SSO портала (раздел 7d).
- Масштаб: **корп-контур** (закрытая зона портала), а не песочница.
