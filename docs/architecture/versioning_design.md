# Архитектура версионирования данных и сценариев

> Статус: рабочее описание (в проработке). Источник фактов о коде — ветка
> `feature/dwh-bb8`. Источник фактов о цикличности обновления — планировщик
> (зафиксировано 2026-06-24).

## 1. Назначение

Описать, как версионируются входные данные симуляции и как поверх них живут
**сценарные прогоны**. Цель — единая модель версий, примиряющая три разных
механизма версионирования источников и сценарные оверрайды.

## 5. Сценарная значимость источников (ключевой вывод)

Принципиально: **«редкие» мастер-источники (#4–#7) — не статичные справочники, а
именно те данные, что меняет планировщик в движке сценарных прогнозов.** Их нельзя
версионировать только по дате — они должны участвовать в измерении `version_id`.

**Сценарно-изменяемые параметры:**

- **Program_heli (#4) и Program (#5):** данные программ полётов/эксплуатации правятся
  **вручную в движке сценарных прогнозов** (объёмы, программа по бортам/типам).
  Program_AC (#3) сюда **не входит** — он из AMOS DWH.
- **MD_Сomponents (#6):** сценарно-значимы — **цены**, **сроки ремонта**, **объём
  ремонта** (макс. количество одновременно в ремонте), **НР и МРР**, **наработка на
  отказ**, **правила пополнения фонда**.
- **Economics (#7):** экономические параметры нужны для сценарного подхода.

То есть сценарный оверрайд затрагивает **все** классы источников, включая мастер-слой,
а не только скаляры прогона.

## 6. Следствия для модели версий (то, что требуется спроектировать)

Из разделов 2–5 вытекают факты-напряжения, которые целевая модель версий должна снять:

1. **`version_id` сегодня — голый ручной счётчик** без реестра и семантики (что значит
   «2», какой пресет, какие оверрайды). Нужен слой-манифест, присваивающий и
   описывающий `version_id`.

2. **Ключ версии не единообразен по таблицам:**
   - #1–#3 — полный срез `(version_date, version_id)`;
   - #4–#5 (flight-программы) — **только `version_date`**, `version_id` в ключе нет
     (`program_ac_direct_loader.py:525`);
   - #6 — версии нет;
   - #7 — собственная ось `effective_date`.
   Сценарные оверрайды Program/MD/Economics требуют, чтобы `version_id` стал сквозным
   ключом и для #4–#7, иначе сценарные варианты этих данных негде хранить раздельно.

3. **Мастер-слой (#6, #7) нужно сделать версионируемым по `version_id`.** Сейчас все
   прогоны на одну дату видят один и тот же `md_components`/Economics. Для сценариев
   (другие цены, сроки/объём ремонта, НР/МРР, экономика) оверрайд должен цепляться к
   `version_id`, а не к общему файлу.

4. **Рассинхрон осей дат.** DWH-снапшот (#1–#3) живёт по календарю AMOS; Program (#4–#5)
   — 2 раза в год; MD/Economics (#6–#7) — несколько раз в год. Модель версий должна
   зафиксировать, что именно склеивает эти разные циклы в один `version_date` прогона
   (сейчас совпадение значений держится вручную).

## 7. Целевая архитектура: всё в корп-DWH, ночные прогоны, слой прогноз-vs-факт

Целевое состояние (зафиксировано планировщиком 2026-06-24):

### 7.1. Полная консолидация в корп-DWH
Все входные и выходные данные в итоге грузятся в **корпоративный DWH**. Excel-источники
(#4, #5) и мастер-слой (#6, #7) мигрируют туда же вслед за #1–#3.

### 7.2. Ночной штатный прогон (`version_id=1`)
- **Каждую ночь автоматически генерируется `version_id=1`** и сохраняется в DWH.
- Берёт **последние известные значения всех таблиц** (latest-known срез источников).
- Это «дефолтный на тот момент» прогон — baseline без сценарных оверрайдов.

### 7.3. Продляющийся прогноз (rolling horizon)
- С момента первой симуляции каждая новая дата:
  - **накапливает чейнджи** новых симуляций выходных таблиц;
  - **добавляет один день в конец** горизонта прогноза.
- Формируется **слой продляющихся и изменяющихся прогнозов** — не перезапись, а
  накопление: каждый прогон фиксируется как срез, горизонт растёт на +1 день/дату.

### 7.4. Трёхмерная матрица прогноз vs накапливающийся факт
- Прогнозы образуют **трёхмерную матрицу** (ось дат прогона × ось дней прогноза ×
  предметные показатели) — «продляющиеся и изменяющиеся прогнозы».
- Параллельно в DWH **накапливаются таблицы факта по трём разрезам**. Три разреза —
  это **AMOS-аналоги #1–#3**: Status_Components, Status_Overhaul, Program_AC. То есть
  факт собирается ровно по тем трём потокам, что уже едут из AMOS DWH.
- Сопоставление этих двух слоёв (прогноз vs факт) — основной аналитический продукт.

### 7.5. Неизменяемость и слой хранения
- Выгрузки прогнозов и факта **хранятся постоянно и не подвергаются изменениям**
  (immutable, append-only) — это и есть основа для исторического сравнения.
- Предполагаемый слой хранения в DWH — **`reports`** (там же, где `reports.amos_heli_*`).

### 7.6. BI: Superset
- Superset BI **уже подключён** и **умеет сравнивать версии по датам**.
- **Ограничение:** сравнение **по `version_id` пока не поддерживается** — Superset
  различает срезы только по дате. Это разрыв: сценарные варианты (`version_id` 2,3…)
  на одну дату в BI пока неразличимы.

### 7.7. Назначение — bigdata для ML/GNN
Накопленный слой прогноз-vs-факт = **подробная bigdata** для:
- сравнения прогноза и факта во времени;
- обучения моделей, включая **графовые нейросети (GNN)** и **машинное обучение**.

То есть версионирование `(version_date, version_id)` — это не только идемпотентность
прогонов, но и **фундамент датасета для ML**: каждая версия — точка в матрице, которую
позже сравнивают с фактом.

## 8. Слой пользовательских сценариев

В дополнение к ночному baseline (`version_id=1`) — отдельный слой сценарных прогонов,
запускаемых планировщиком.

### 8.1. Хранение — слой `staging`
Пользовательские сценарии сохраняются в таблицах слоя **`staging`** (отдельно от
фактового/прогнозного слоя `reports`).

### 8.2. Самодостаточный срез: исходники + результаты под одним `version_id`
Под **одним `version_id` (= 2 и более)** сохраняются **вместе**:
- все **исходные данные** прогона (срез всех источников на момент запуска);
- **результаты** прогноза.

То есть срез сценария **самодостаточен и воспроизводим** — данные не разъезжаются с
результатом.

### 8.3. Авторство и доступ — Keycloak
- Сохраняется **логин автора прогона из Keycloak** вместе со срезом.
- **Доступ в движок сценариев — только через Keycloak** (анонимных прогонов нет).

### 8.4. Типы сценарных условий
- **По характеру задания:** однократные (точечные) **или** диапазонные по показателям.
- **Условие достижения параметра:** `не выше` / `не ниже` / `точно равно`.
- **Комбинация условий:** по логике **И** и **ИЛИ**.

### 8.5. Нумерация `version_id` — выделенный диапазон
- `version_id = 1` — ночной baseline (раздел 7.2).
- **Сценарным прогнозам выделяется четырёхзначный диапазон, например `2000–2999`.**

### 8.6. Хранение и ретеншн
- Политика хранения staging-таблиц сценариев — **определить в течение года**.
- Требование: возможность **выгрузить исходные данные сценария в Excel**.
